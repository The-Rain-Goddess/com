Introduction

The PowerManager Digital Control Platform (PM-DCP) is an electronic module for controlling Generac Power Systems equipment. This module has both RS-232 and RS-485 communications ports which can be used to share data with the outside world. Port functionality is configurable such that different ports may be configured for either hardware standard and may be used for connecting a touchscreen display, a remote annunciator, a telephone line, Modbus control devices, and/or computers running "GenLink-DCP" (with some limitations on the number of simultaneous connections). If the particular installation is a linked, multi-generator site, then data about each generator can be obtained from the master controller. Examples are described later in this document.

The method used to communicate with all PM-DCP controllers is typically the industry standard Modbus protocol. (The touch-screen display uses a proprietary protocol.) Details about this protocol may be obtained from various sources including:

http://www.modbus.org/

Generac produces a product known as "GenLink-DCP" which is a PC based application designed to allow communication with individual generators or with multiple linked generators via the master controller. The physical connection between GenLink-DCP and the generators may be a direct serial link or it can be via modem.

This application uses the Modbus protocol to continuously monitor and display the state of each generator and master controller. GenLink-DCP can also be used to change the operating parameters of the generators and of the master controller.

It isn't necessary to use GenLink-DCP as the basis for communication with the PM-DCP controllers. Since Modbus has been chosen as the standard protocol, any piece of equipment acting as a Modbus "master" can be used for data exchange.

Modbus Protocol

Modbus controllers can be set up to communicate in one of two modes: ASCII or RTU. The PM-DCP series controllers will only work in RTU mode, that is, communications are done as eight-bit binary data and with shorter timeout values.

The function, protocol, speed, etc. are programmable for each port, with the following defaults.

The PM-DCP controllers act as slaves in a master-slave configuration and depending upon the installation, many slaves can be linked together to one master. In order to differentiate units from each other, each unit is assigned an address unique to that link, even if there is only one unit. These addresses are programmed into the slave units using GenLink-DCP while directly connected to the controller.

Data is stored and accessed from Modbus "registers" and "files" within the PM-DCP. Registers, and ranges of registers or file addresses contain specific data such as RPM, Hz, Volts, etc. and can be accessed or changed as defined by the Modbus protocol.

Generator (Slave) Addressing

In a single generator installation it may be unnecessary to program a slave node address into the controller because each controller will always respond to the special address, 250. This is done as a convenience, and to support simple devices such as the remote annunciator which expect to be used on a dedicated point-to-point connection.

The special node address 250 is not a valid Modbus node address. For multi-generator sites, or where industry-standard Modbus is expected, each PM-DCP should be assigned a valid Modbus slave node address in the range from 0 through 247, inclusive.

The default slave node address found in PM-DCP controls is typically 100. In case the node address is changed and forgotten, the GenLink-DCP application can scan all possible node addresses until a PM-DCP control responds.

In a multiple generator installation, each generator must have a unique node address. This can be configured using GenLink-DCP while directly connected.

The master controller should have a slave node address of its own, for connection to a building network for example, but will also need to be programmed with the slave node addresses of each of the generator sets which it manages.

Setting Up Communication Parameters

The speed of communication (bit rate), number of stop bits and parity are all programmable using GenLink-DCP. These should be set up to match each other and the Modbus master (or GenLink-DCP itself). We suggest using the default settings for both the Modbus ports of the PM-DCP:

9600[bps], 8 data bits, no parity, 1 stop bit.

or, if the remote annunciator is connected:

4800[bps], 8 data bits, no parity, 1 stop bit.

(Note that the default number of stop bits is different from that of the older "F Panel" controller.)

Modbus Specifics

The PM-DCP communicates in Modbus "RTU mode", which is an eight-bit binary format.

(Note: There is a Modbus "ASCII mode" which consists of hexadecimal digits encoded as ASCII characters and which requires much longer timeout values. ASCII mode is not supported due to its slower effective data rate and especially because its overly-long timeout requirements could block other communications.)

Two groups of Modbus functions are supported for full access to all of the PM-DCP's parameters: register functions and file-record functions.

Virtual Register Mapping

The PM-DCP supports two of the data access modes defined by the Modbus protocol: "register" access and "file record" access. Since the register access space wasn't large enough to accommodate all the configurable parameters of the PM-DCP, the two access modes have been mapped onto a "flat" address space called the "virtual register map". Use of these virtual addresses is encouraged, for portability to other protocols in the future.

Virtual Registers

The PM-DCP has over two hundred parameters defined for configuration settings and displayable information. It is also expected to act as a pass-through interface to more than ten other PM-DCP-controlled devices.

Mapping interface functions to "registers" is a well-established standard for control systems. Human Interface Devices (HIDs) from most vendors support some form of register-based protocol and the same principle is used in building-management systems and public utility remote monitoring and control systems (SCADA).

As in the Modbus protocol, multi-byte values are expected to be "Big-Endian"". That is, "most significant byte first". This applies to two-byte values as well as to larger numeric values such as UINT32 which is four bytes. (It does not apply to character strings.) Thus, a four-byte value of 0x12345678 which is expected to begin at register address 0x0100 would be stored like so:

Register-based protocols typically have commands to read and write single addresses or address ranges in a virtual, addressable memory. By using this scheme, we provide a foundation for communication protocols and a uniform inter-module interface within the PM-DCP. We recommend that customers who wish to write their own applications to interface with the PM-DCP should do the same. For setting up just a few registers, or for using a third-party Modbus device, the reference material in this document should suffice.

Pass-Through to Modbus

Because many of these register-based serial protocols don't allow multiple "master" devices on the same bus, we provide a way to access other devices connected to a PM-DCP unit via Modbus. This is useful for configuring multiple PM-DCP devices through a single connection, or allowing a building-management system to determine the status of multiple generators coordinated by a PM-DCP-based PowerManager. It also provides a way to seamlessly "expand" the PM-DCP's capabilities simply by attaching Modbus devices to it. To the Modbus interface, they will all appear to be a single device.

Since we don't anticipate the need to pass-through communication to any more than a dozen devices, we use the uppermost four bits of virtual address space to specify which device is being addressed.

If these device bits are zero, the current device (that specified by the Modbus slave node number) is addressed. Otherwise, the PM-DCP will pass the read/write request through to the slave node with the address given in these bits on it's other Modbus link.

This pass-through functionality, when implemented within the virtualized register API, gives the PM-DCP surprising flexibility:

  • Most HIDs aren't capable of fetching values simultaneously from multiple devices. Register mapping makes the PM-DCP and its managed devices all appear to be just a single device.
  • The PM-DCP can act as a protocol converter, allowing access to Modbus slaves even though the master is being connected to with another protocol (like Siemens-PPI, or any protocol added to the PM-DCP in the future).
  • Other Modbus devices can be accessed the same way, almost transparently.

Addressing Virtual Registers

There are two useful ways of looking at the virtual address space; as a "memory map" showing blocks of allocated space, and as a set of bit fields within the 32-bit address. Since the layout of this address space has been heavily influenced by the capabilities of the standard Modbus commands for "register" and "file record" commands, the memory map has gaps which will only make sense when one considers how the address bits map into the Modbus protocol.

Mapping Virtual Registers to Modbus

Two main forms of data transfer operations are used when communicating with the PM-DCP platform via Modbus: "register" and "file record" reads and writes. The former are universally understood by any device which handles the protocol. "File record" operations, on the other hand, are not as common.

Since the Modbus pass-through functionality has to exist within the address spaces available to Modbus, the uppermost four bits of either space will be used for the pass-through node address. That means bits 15..12 of either the register address or the file number. (Zero indicates the device itself, rather than a pass-through). These four bits will need to be shifted left to fit into bits 31..28 for the virtual address scheme.

For register access, the remaining twelve bits of register address are shifted left one bit. This aligns the Modbus word-addressing with the virtual byte-addressing.

File record access begins likewise by putting bits 15..12 of the file number into virtual address bits 31..28. Then the lowest nine bits of the file number will be put into virtual address bits 27..19. (File number bits 11..9 are unused and must be set to zero.) The lowest thirteen bits of the "record number" are put into virtual address bits 18..6, leaving virtual address bits 5..0 to address bytes in the file record. Each file record is thus 64 bytes wide.

File record access functions for Modbus pass-through would require adjustment to 64-byte boundaries, and is discouraged.
